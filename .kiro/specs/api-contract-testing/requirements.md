# Requirements Document

## Introduction

This feature introduces consumer-driven contract testing between the Nova Rewards frontend (consumer) and backend (provider) using the Pact framework. Contract tests capture the exact API interactions the frontend depends on and verify that the backend continues to honour those interactions. The goal is to detect breaking API changes before they reach integration testing or production, with results visible in a Pact Broker dashboard and contract failures blocking PR merges.

## Glossary

- **Pact**: An open-source consumer-driven contract testing framework that records API interactions as JSON contract files ("pacts").
- **Consumer**: The Nova Rewards frontend application that makes HTTP requests to the backend API.
- **Provider**: The Nova Rewards backend API that responds to HTTP requests from the Consumer.
- **Pact_Broker**: A centralised server that stores, versions, and shares pact files between Consumer and Provider CI pipelines.
- **Contract**: A JSON file generated by the Consumer test suite that describes the HTTP interactions the Consumer expects from the Provider.
- **Consumer_Test**: A frontend unit test that uses a Pact mock server to record an expected interaction and generate a Contract.
- **Provider_Verification**: A backend test run that replays each interaction from a published Contract against the live Provider and asserts the responses match.
- **Can_I_Deploy**: A Pact Broker CLI command that queries whether a Consumer or Provider version is safe to deploy based on verified contracts.
- **Webhook**: An HTTP callback configured in the Pact_Broker that triggers Provider_Verification when a new Contract is published.
- **CI_Pipeline**: The GitHub Actions workflow that builds, tests, and validates the codebase on every pull request and push.
- **Breaking_Change**: Any modification to a Provider response or request schema that causes at least one existing Consumer_Test interaction to fail verification.

## Requirements

### Requirement 1: Consumer Contract Generation

**User Story:** As a frontend developer, I want Pact consumer tests for every backend API call, so that any API change that breaks the frontend is detected automatically.

#### Acceptance Criteria

1. THE Consumer_Test suite SHALL cover all API endpoint groups defined in the OpenAPI specification: Auth (`/auth/login`, `/auth/register`), Transactions (`/transactions/record`, `/transactions/merchant-totals`), Campaigns (`/campaigns`, `/campaigns/{merchantId}`), Rewards (`/rewards`, `/rewards/{id}`), Admin (`/admin/stats`, `/admin/users`, `/admin/rewards`), Leaderboard, Drops, and User profile endpoints.
2. WHEN a Consumer_Test executes, THE Consumer_Test SHALL use a Pact mock server to intercept HTTP requests and record the expected request method, path, headers, query parameters, and response body as an interaction in a Contract file.
3. WHEN a Consumer_Test records an interaction, THE Consumer_Test SHALL assert that the response received from the mock server matches the expected response defined in the interaction.
4. WHEN all Consumer_Tests in a test run complete, THE Consumer_Test suite SHALL produce one Contract file per Consumer–Provider pair in the `pacts/` output directory.
5. IF a Consumer_Test fails to match the mock server response, THEN THE Consumer_Test suite SHALL exit with a non-zero status code and report the mismatched interaction.
6. THE Consumer_Test suite SHALL cover both success (2xx) and error (4xx) response scenarios for each endpoint.

---

### Requirement 2: Contract Publication to Pact Broker

**User Story:** As a developer, I want generated contracts published to a central Pact Broker, so that the Provider team can access and verify them without manual file sharing.

#### Acceptance Criteria

1. WHEN the Consumer CI job completes successfully, THE CI_Pipeline SHALL publish all Contract files in the `pacts/` directory to the Pact_Broker using the Pact CLI.
2. WHEN publishing a Contract, THE CI_Pipeline SHALL tag the published Contract with the Consumer's Git branch name and commit SHA.
3. THE Pact_Broker SHALL store each published Contract version and retain the full version history for the Consumer–Provider pair.
4. IF the Pact_Broker is unreachable during publication, THEN THE CI_Pipeline SHALL fail the publish step and report the connection error.
5. THE Pact_Broker SHALL be configured with authentication credentials so that only authorised CI jobs can publish or retrieve contracts.

---

### Requirement 3: Provider Verification

**User Story:** As a backend developer, I want the backend CI to verify all published consumer contracts, so that breaking API changes are caught before merging.

#### Acceptance Criteria

1. WHEN the Provider CI job runs, THE Provider_Verification step SHALL fetch all Contract files tagged for the current branch from the Pact_Broker and replay each recorded interaction against the running Provider.
2. WHEN replaying an interaction, THE Provider_Verification step SHALL assert that the Provider response status code, headers, and body match the expectations recorded in the Contract within a tolerance of flexible body matching (type and structure, not exact values).
3. IF any interaction fails verification, THEN THE Provider_Verification step SHALL report the failing interaction name, the expected response, and the actual response, and exit with a non-zero status code.
4. WHEN Provider_Verification completes, THE Provider_Verification step SHALL publish the verification result (pass or fail) back to the Pact_Broker, tagged with the Provider's Git branch name and commit SHA.
5. THE Provider_Verification step SHALL run after unit tests and before integration tests in the CI_Pipeline job order.
6. WHEN the Pact_Broker receives a new Contract from the Consumer, THE Pact_Broker SHALL trigger a Webhook to initiate Provider_Verification on the Provider's CI pipeline within 5 minutes.

---

### Requirement 4: Breaking Change Gate on Pull Requests

**User Story:** As a team lead, I want breaking contract changes to block PR merges automatically, so that no API change silently breaks the frontend.

#### Acceptance Criteria

1. WHEN a pull request targets the `main` or `develop` branch, THE CI_Pipeline SHALL execute the Can_I_Deploy check for the Consumer version before allowing the PR to be merged.
2. WHEN a pull request targets the `main` or `develop` branch, THE CI_Pipeline SHALL execute the Can_I_Deploy check for the Provider version before allowing the PR to be merged.
3. IF the Can_I_Deploy check returns a failure for either Consumer or Provider, THEN THE CI_Pipeline SHALL set the GitHub status check to failed and block the PR merge.
4. WHEN the Can_I_Deploy check passes for both Consumer and Provider, THE CI_Pipeline SHALL set the GitHub status check to success and permit the PR merge.
5. THE Can_I_Deploy check SHALL query the Pact_Broker for the verification status of the specific Consumer and Provider versions being deployed together.

---

### Requirement 5: Pact Broker Dashboard Visibility

**User Story:** As a developer or QA engineer, I want contract test results visible in the Pact Broker dashboard, so that I can inspect the status of each Consumer–Provider interaction at any time.

#### Acceptance Criteria

1. THE Pact_Broker SHALL provide a web dashboard that displays the verification status (pass, fail, or pending) for each Consumer–Provider Contract version.
2. WHEN a Provider_Verification result is published, THE Pact_Broker SHALL update the dashboard to reflect the new verification status within 60 seconds.
3. THE Pact_Broker dashboard SHALL display the Consumer version tag, Provider version tag, verification date, and a list of individual interaction results for each Contract.
4. WHEN an interaction fails verification, THE Pact_Broker dashboard SHALL display the expected and actual response diff for that interaction.
5. THE Pact_Broker SHALL retain verification history for a minimum of 90 days so that teams can audit past contract states.

---

### Requirement 6: CI Pipeline Ordering and Isolation

**User Story:** As a DevOps engineer, I want contract tests to run in the correct CI stage and in isolation from integration tests, so that failures are attributed to the right cause.

#### Acceptance Criteria

1. THE CI_Pipeline SHALL execute Consumer_Tests as a dedicated job that runs after the frontend build job and before any integration test job.
2. THE CI_Pipeline SHALL execute Provider_Verification as a dedicated job that runs after the backend unit test job and before any integration test job.
3. WHEN the Consumer_Test job fails, THE CI_Pipeline SHALL skip the Contract publication step and mark the overall pipeline as failed.
4. WHEN the Provider_Verification job fails, THE CI_Pipeline SHALL mark the overall pipeline as failed without cancelling unrelated jobs (e.g., frontend build).
5. THE Consumer_Test job and the Provider_Verification job SHALL each complete within 10 minutes under normal conditions.
6. THE Consumer_Test job SHALL run in an isolated environment with no network access to the real Provider, relying solely on the Pact mock server.
